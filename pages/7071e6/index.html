                    # format

                    --pretty=format 选项允许你指定如何格式化每个提交的输出。下面是一些常用的 --pretty=format 选项:

                    • %H: 提交对象(commit)的完整哈希值
                    • %h: 提交对象的简短哈希值
                    • %T: 树对象(tree)的完整哈希值
                    • %t: 树对象的简短哈希值
                    • %P: 父对象(parent)的完整哈希值
                    • %p: 父对象的简短哈希值
                    • %an: 作者(author)的名字
                    • %ae: 作者的电子邮件地址
                    • %ad: 作者修订日期(可以通过 --date 选项进行格式化)
                    • %ar: 作者修订日期，按多久以前的方式显示
                    • %cn: 提交者(committer)的名字
                    • %ce: 提交者的电子邮件地址
                    • %cd: 提交日期
                    • %cr: 提交日期，按多久以前的方式显示
                    • %s: 提交说明(commit message)
                    • %b: 提交说明正文
                    • %N: 提交的分支名称
                    • %d: ref 名称(包括分支、标签等)
                    • %gD: reflog Selector, e.g., refs/stash@{0}
                    • %gd: shortened reflog Selector, e.g., stash@{0}
                    • %gs: reflog Subject

                    你可以通过组合这些选项来自定义 git log 的输出。例如:

                    git log --pretty=format:'%h - %an, %ar : %s'
                    1

                    这将显示简短的提交哈希值、作者名字、相对日期和提交说明，格式如下:

                    e3d1e7a - Alice, 2 days ago : Fixed bug in authentication
a1b2c3d - Bob, 3 weeks ago : Added new feature
                    1
                    2

                    结合 --graph 选项，可以更直观地查看提交历史的分支和合并情况。

                    git log --pretty=format:'%h %s' --graph
                    1

                    例如，输出可能类似于:

                    * e3d1e7a Fixed bug in authentication
| * a1b2c3d Added new feature
|/
* 8b1f3d7 Initial commit
                    1
                    2
                    3
                    4

                    这有助于更好地理解项目的分支结构和合并历史。

                    # show

                    link (opens new window)

                    • git show <sha>/<tag>: 查看某一个节点的修改内容
                    • git show HEAD@{5}: 你可以使用 @{n} 来引用 reflog 中输出的提交记录

                    merge commit

                    When you use git show on a merge commit, it shows the commit message and metadata but does not typically show a diff of changes because the changes are already present in the commits being merged.

                    If you want to see the actual changes that were merged, you can look at the individual commits that were merged. To see the changes introduced by the merge commit, you can use:

                    git log -p -1 74707fa8a985b9059a7663c539df1efcc279eeb4
                    1

                    or

                    git diff 2cefc417 f8fcfff4
                    1

                    Here's what these commands do:

                    • git log -p -1 <commit>: Shows the patch introduced in the commit, even for merge commits.
                    • git diff <parent1> <parent2>: Shows the differences between the two parent commits of the merge.

                    The git diff command will give you the actual changes that were combined by the merge commit.

                    # stash

                    git stash 是 Git 版本控制系统中的一个命令, 用于暂时保存当前工作目录的修改, 并将其放入一个"存储堆栈"中。这样可以让您在不提交修改的情况下切换到其他分支或者进行其他操作。

                    当您在工作目录中有一些修改, 但又不想立即提交这些修改时, 可以使用 git stash 命令将这些修改保存起来。这将会清空当前工作目录, 使其恢复到上一次提交的状态。

                    常用的命令:
                    • git stash: 将当前工作目录的修改保存到一个临时的存储堆栈中。您可以继续在干净的工作目录上进行其他操作
                    • git stash save <message>: 将当前的工作目录和暂存区的更改保存到堆栈中, 并添加一个描述信息。这是最常用的形式。
                    • git stash list: 列出当前的所有存储的工作目录状态。每个存储的状态都有一个唯一的索引(如 stash@{0}、stash@{1} 等)。
                    • git stash apply: 应用最新的存储状态, 但不从堆栈中移除它。
                    • git stash pop: 应用最新的存储状态并从堆栈中移除它。
                    • git stash drop stash@{n}: 移除堆栈中指定索引的存储状态。
                    • git stash clear: 清空堆栈, 移除所有存储状态。
                    • git stash branch new-branch-name: 创建新分支并将堆栈中的最新存储状态应用到新分支上, 然后清空堆栈。

                    # rev-list

                    --ancestry-path

                    git rev-list 的 --ancestry-path 参数用于限制列出的提交历史，使其只包括从一个特定提交到另一个提交之间的路径上的提交。这个选项特别有用在需要找到两个提交之间的所有提交时，但只关心它们在同一个祖先路径上的情况。

                    使用场景:

                    • 找到分支上的提交: 可以用来找到某个分支上的所有提交，而不包括其他分支上的提交。
                    • 调试和审查历史: 在复杂的提交历史中，帮助用户理解两个提交之间的直接祖先路径上的提交情况。
                    结合其他参数

                    --ancestry-path 可以与 --graph、--oneline 等其他选项结合使用，以获得更好的可读性和显示效果。

                    示例:

                    git rev-list --ancestry-path --graph --oneline <commitA>..<commitB>
                    1

                    # 撤销提交到 stage 的修改

                    • git reset HEAD~ 将导致你的本地工作区、暂存区和提交历史都被重置到了前一个提交的状态. 也就是完全放弃了本地的修改, 一定要非常小心这个命令的使用
                    • git reset --hard HEAD~ 这将删除最近的提交并重置工作目录和暂存区为上一个提交的状态
                    • git revert HEAD 将创建一个新的提交, 撤销最近的提交所做的更改。这种方法不会删除之前的提交历史, 而是创建一个新的提交来撤销之前的提交。

                    # 查看一个修改什么时候合并到master的

                    Git 查看指定 commit 何时 merge 进指定分支 (opens new window)

                    [alias]
    find-merge = "!sh -c 'commit=$0 && branch=${1:-HEAD} && (git rev-list $commit..$branch --ancestry-path | cat -n; git rev-list $commit..$branch --first-parent | cat -n) | sort -k2 -s | uniq -f1 -d | sort -n | tail -1 | cut -f2'"
    show-merge = "!sh -c 'merge=$(git find-merge $0 $1) && [ -n \"$merge\" ] && git show $merge'"
                    1
                    2
                    3
                    find-merge 别名

                    这个别名用于找到两个分支之间的最新合并提交。以下是它的工作原理:

                    • commit=$0: 将第一个参数(作为起始提交)赋值给变量 commit。
                    • branch=${1:-HEAD}: 将第二个参数(作为目标分支)赋值给变量 branch，如果未提供则默认为 HEAD。
                    • git rev-list $commit..$branch --ancestry-path: 列出从起始提交到目标分支的所有提交(包括所有路径)。
                    • git rev-list $commit..$branch --first-parent: 列出从起始提交到目标分支的所有提交(仅包括第一父路径)。
                    • 使用 cat -n 为每个提交编号，并将两次 rev-list 的结果合并。
                    • 使用 sort -k2 -s 按提交哈希进行排序。
                    • 使用 uniq -f1 -d 找到重复的提交(即两个分支共有的提交)。
                    • 使用 sort -n 按编号排序。
                    • 使用 tail -1 取最后一个提交(即最新的合并提交)。
                    • 使用 cut -f2 提取提交哈希。

                    最终，它返回的是两个分支之间的最新合并提交的哈希。

                    show-merge 别名

                    这个别名用于显示两个分支之间的最新合并提交。以下是它的工作原理:

                    • merge=$(git find-merge $0 $1): 调用 find-merge 别名并获取合并提交的哈希。
                    • [ -n \"$merge\" ]: 检查合并提交哈希是否为空。
                    • git show $merge: 如果合并提交哈希不为空，则显示该合并提交的详细信息。

                    # 科学的忽略本地mock文件

                    在git项目中，假设有一个文件 src/mock.tsx是我本地的mock文件，在任何情况下，我都不希望提交到remote， 怎么实现这个行为，防止误提交。 git hooks？

                    为了确保 src/mock.tsx 文件在任何情况下都不会被提交到远程仓库，你可以采取以下几种方法:

                    1. 使用 .gitignore 文件:

                    在你的项目根目录下的 .gitignore 文件中添加以下行:

                    src/mock.tsx
                    1

                    这样，Git 会忽略这个文件的变化，不会将其包括在提交中。

                    1. 使用 Git hooks:

                    你可以使用 Git 的 pre-commit 钩子来防止该文件被提交。Git hooks 是在特定事件发生时自动运行的脚本。你可以创建一个 pre-commit 钩子来检查是否试图提交了 src/mock.tsx 文件，并在检测到时中止提交。

                    首先，确保你的项目中有一个 .git/hooks 目录(默认存在)。然后创建一个名为 pre-commit 的文件，并使其可执行:

                    touch .git/hooks/pre-commit
chmod +x .git/hooks/pre-commit
                    1
                    2

                    编辑 pre-commit 文件，添加以下内容:

                    #!/bin/sh
if git diff --cached --name-only | grep -q 'src/mock.tsx'; then
  echo "Error: Attempt to commit src/mock.tsx"
  exit 1
fi
                    1
                    2
                    3
                    4
                    5

                    The command git diff --cached --name-only | grep -q 'src/mock.tsx' is used to check if the file src/mock.tsx is staged (cached) for commit in your Git repository. The -q option in grep suppresses output, and the command will return a success exit code (0) if the file is found, or a failure exit code (1) if it is not found.

                    这个脚本会在每次提交之前运行，检查暂存区中是否包含 src/mock.tsx 文件。如果包含，则中止提交并显示错误信息。

                    1. 使用 Git 属性和 update-index:

                    如果你已经不小心将 src/mock.tsx 添加到版本控制中，可以使用以下方法:

                    首先将其从版本控制中移除，但保留在本地工作目录中:

                    git rm --cached src/mock.tsx
                    1

                    然后将其添加到 .gitignore 中:

                    src/mock.tsx
                    1

                    最后，确保它不会再次被意外添加:

                    git update-index --assume-unchanged src/mock.tsx
                    1

                    这样，即使不小心添加了 src/mock.tsx，Git 也会忽略它的变化。

                    以上方法都可以有效防止 src/mock.tsx 文件被提交到远程仓库，具体选择哪种方法可以根据你的实际需求和团队工作流程来决定。

                    # stage changes(暂存区)

                    在 Git 中, "stage changes"指的是将修改或新增的文件添加到暂存区(也称为索引)的过程。暂存区是在提交更改之前的一个中间区域, 用于准备提交到版本控制系统中的文件。

                    当你在工作目录中修改或创建文件后, 这些更改不会立即被记录到 Git 历史中。相反, 你需要使用 git add 命令将这些更改添加到暂存区。通过将更改文件添加到暂存区, 你告诉 Git 将这些更改包含在下一次提交中。

                    可以通过以下步骤将更改添加到暂存区:

                    1. 使用git add命令加入要暂存的文件。例如, git add myfile.txt 将myfile.txt文件添加到暂存区。
                    2. 可以使用 git rm --cached <file> 将文件移出暂存区
                    3. 使用git status命令检查已暂存的更改。暂存的更改将显示在"Changes to be committed"部分中。 一旦你将更改添加到暂存区, 你可以使用git commit命令将其提交到 Git 历史记录中。提交后, 暂存区中的更改将成为一个新的版本, 并且可以在需要时回滚或查看历史记录。

                    通过将更改分为工作目录、暂存区和历史记录这三个区域, Git 提供了一种灵活而强大的版本控制系统, 使你能够精确控制文件的修改和提交过程。

                    # 什么是第一父提交

                    什么是 Git 父提交 (opens new window)。 祖父引用 (opens new window)

                    合并:

                    合并分支时产生的提交有两个父提交，第一父提交为合并前当前分支的最后一次提交，第二父提交为被合并的分支的最后一次提交。 合并提交的第一父提交是你合并时所在分支(通常为 master)，而第二父提交是你所合并的分支(例如 feature)

                    # 祖先引用

                    • ^: 引用的尾部加上一个. Git 会将其解析为该引用的上一个提交
                    • ~: 另一种指明祖先提交的方法。因此 HEAD~ 和 HEAD^ 是等价的。 而区别在于你在后面加数字的时候。 HEAD~2 代表"第一父提交的第一父提交"，也就是"祖父提交"

                    如何相对运动？

                    • master^ 相当于 master 的父节点。
                    • master^^ 是 master 的第二个父节点
                    • git checkout HEAD~4 向后移动 4 位
